# お返し品の品目カテゴリ提案 設計

作成日: 2026-06-17
対象機能: お返し品の提案画面（`frontend/src/App.tsx` の `suggest` 画面 + `backend/app/catalog/*`）
関連: [[return-gift-suggestion-design]]（楽天アフィリエイト・日次バッチ厳選DBの正本設計）

## 背景と課題（見極め）

ユーザーから3つの声が挙がった。

1. 一覧にいい商品がなかったときにさらに表示されるといい
2. 食料品とかしか表示されないときがある。食料品以外もみたい
3. 更に検索、みたいなのができるといい

3つは別々の要望に見えるが、根は1つ。「提案が外れたとき、自分でハンドルを握り直す手段がない」。

現状の画面は、1イベント（相手の続柄 × 用途 × 予算）に対して、サーバが「用途スラッグ × 価格帯」の1バケツだけを引き、厳選トップ最大10件を続柄フィットで並べて返すだけ（`backend/app/catalog/adapter.py` の `DynamoCatalogAdapter.suggest`）。フロントは受け取った10件をそのまま並べる（`App.tsx` の `screen === "suggest"`）。「もっと見る」も、カテゴリ切替も、検索もない take-it-or-leave-it のリストになっている。

しかもバケツは用途（内祝い・香典返し等）で組まれていて、楽天ジャンルの売れ筋がそのまま食品・消耗品に寄るため「食料品ばかり」に見えやすい。一方、ピックを断るユーザーは用途ではなく品目（タオルがいい、カタログがいい、食品以外）で考えている。この「システムは用途で、ユーザーは品目で考えている」ズレが、箱に閉じ込められた感覚を生んでいる。

したがって作るべきは「検索を足す・カテゴリを足す・ページングを足す」3つのパッチではなく、厳選ピックから自分で探す状態へ抜ける、ひとつの一貫した逃げ道。今回はそれを品目カテゴリで実現する。

## ゴール / 非ゴール

ゴール:
- お返し品の提案画面に、品目カテゴリで絞り込めるタブ（チップ）を追加する。
- 「おすすめ」（現行の用途×予算の厳選）は既定タブとして無改造で残す。
- 非食品の受け皿（タオル・寝具／食器・キッチン／カタログ／日用品）をトーン両方に持ち、「食料品しか出ない」を解消する。
- 品薄で0件のタブは出さない（タップして空、を防ぐ）。
- 既存の挙動・データ・楽天規約対応を壊さない（非破壊の追加）。

非ゴール（今回スコープ外）:
- 自由キーワード検索（声3）。厳選DBの品質を保つ選択として、品目で広げることで代替する。
- 無限ページング／「もっと見る」での件数追加（声1の depth 方向）。各品目バケツは厳選トップ10のまま。breadth（品目の広がり）で代替する。
- 楽天ライブ検索（リクエスト時の実API呼び出し）。日次バッチ厳選DBの方式は維持する。
- 相手個人の購入履歴に基づく除外（v2候補）。

## 品目タクソノミ（トーン別・トレンド裏取り済み）

2026年の内祝い・お返しの人気カテゴリ調査（ギフトモール／三越伊勢丹ムードマーク／Yahoo!ショッピング／おこめやノート 等）で裏取りした。お米（特に出産内祝いの体重米・名入れ米）はグルメ・食品に内包、名入れは品目ではなく横断属性のため独立タブにしない。

トーンは `app/domain/tone.py` の `tone_of(purpose)`（慶事 = それ以外 / 弔事 = mourning）で判定する。

慶事（お祝い返し）— 7タブ:

| cat_slug | 表示名 | 楽天検索キーワード（目安） |
| --- | --- | --- |
| sweets | スイーツ・お菓子 | 内祝い 洋菓子 焼き菓子 |
| gourmet | グルメ・食品 | 内祝い グルメ 食品 |
| drink | 飲料（コーヒー・日本茶・紅茶） | 内祝い コーヒー 日本茶 ギフト |
| towel | タオル・寝具 | 内祝い タオル ギフト |
| tableware | 食器・キッチン | 内祝い 食器 ギフト |
| sake | お酒 | 内祝い 酒 ギフト |
| catalog | カタログギフト | 内祝い カタログギフト |

弔事（香典返し）— 5タブ:

| cat_slug | 表示名 | 楽天検索キーワード（目安） |
| --- | --- | --- |
| drink | 飲料（緑茶・コーヒー） | 香典返し 緑茶 コーヒー |
| food | 食品（海苔・乾物・お米・佃煮） | 香典返し 海苔 乾物 |
| towel | タオル・寝具 | 香典返し タオル |
| daily | 洗剤・日用品 | 香典返し 洗剤 日用品 |
| catalog | カタログギフト | 香典返し カタログギフト |

弔事はスイーツ・お酒・ペア食器など華美な品を出さない。ジャンルID（楽天ランキングAPI用）は実装時に各キーワードの上位最頻ジャンルを採取して確定する（既存 `RAKUTEN_GENRE_BY_CATEGORY` と同じ手順）。

注: 既定タブ「おすすめ」は現行の用途×予算バケツをそのまま使い、品目軸を持たない（横断の総合ベスト）。

## アーキテクチャ

非破壊の「並行追加」。既存の用途×予算バケツ・配信パスは一切触らず、品目バケツ・マニフェスト・任意パラメータを足す。

### データモデル（DynamoDB: NoshiCatalogTable）

- 既存（変更なし）: `PK = BUCKET#<用途slug>#<予算band>`, `SK = RANK#01..10`。
- 新規 品目バケツ: `PK = BUCKET#<tone>#<cat_slug>#<予算band>`, `SK = RANK#01..10`。
  - `<tone>` は `cele`（慶事）/ `mourn`（弔事）。
  - 既存の `store.replace_bucket(slug, band, ...)` / `read_bucket(slug, band, ...)` は `bucket_pk(slug, band) = f"BUCKET#{slug}#{band}"` を使うだけなので、`slug` に `f"{tone}#{cat_slug}"` を渡せば PK が `BUCKET#cele#towel#5000-9999` の形になり、store は無改造で流用できる。
- 新規 マニフェスト: `PK = MANIFEST#<tone>#<予算band>`, `SK = MANIFEST`, 属性 `categories`（その (tone, band) で在庫のある cat_slug の順序つきリスト）。バッチが各 (tone, band) の処理後に「トップ10が1件以上入った品目」を集めて書く。配信時に1件読むだけで、出すべきチップが分かる（空タブを出さない）。マニフェストにも `expiresAt`（48h）を付ける。

バケツ数: 既存 9用途 × 7予算 = 63。新規 (7慶事 + 5弔事) 品目 × 7予算 = 84。合計 147（約2.3x）。マニフェストは (2トーン × 7予算) = 14 件。

### 配信（`backend/app/catalog/adapter.py`）

`DynamoCatalogAdapter.suggest` に任意の `category: str | None` を追加する（既存シグネチャは後方互換のためデフォルト引数で拡張）。

- `category` 無指定 = 現行どおり用途×予算バケツ（おすすめ）。挙動変更なし。
- `category` 指定 = `tone = tone_slug(purpose)` を求め、`slug = f"{tone}#{category}"` で品目バケツを `read_bucket` し、続柄フィットで並べ替え（`_fit_sorted` をそのまま使用）＋隣接価格帯補完（`band_neighbors`）＋鮮度マスク（既存ロジック）。0件なら金額目安フォールバック。
- 併せて available_categories を返す経路を用意する。`suggest` とは別メソド `available_categories(budget, purpose) -> list[{slug,label}]` を追加し、マニフェスト1件を読んでトーン別の表示名つきで返す。サービス層はおすすめ要求時にこれを呼ぶ。

`GiftCatalogPort`（`backend/app/ports.py`）に `category` 引数と `available_categories` を追加する。スタブ実装（テスト用フォールバック）も追従させる。

### API（`backend/app/main.py` / `services.py` / `frontend/src/api.ts`）

- `GET /api/events/{event_id}/suggestions` に任意クエリ `category` を追加。
- レスポンスを `{ "suggestions": [...], "categories": [{slug,label}...] }` に拡張。`categories` はそのイベントのトーン×予算で在庫のある品目（おすすめ要求時も常に返し、フロントがチップを描ける）。`category` 指定時も同じ `categories` を返す（タブの再描画に使える）。
- `services.suggest_returns` は `category` を `catalog.suggest` に素通しし、`catalog.available_categories` の結果を併せて返す。
- フロント `api.suggestions(eventId, budget, relationship, purpose, category?)` に引数追加。`category` は URL クエリに付ける（`encodeURIComponent`）。

### バッチ（`backend/app/catalog/job.py` / `curation.py` / `buckets.py`）

- `buckets.py` に品目定義を追加: `ITEM_CATEGORIES: dict[tone, list[(cat_slug, label, keyword)]]` と `RAKUTEN_GENRE_BY_ITEM_CATEGORY: dict[(tone, cat_slug), str | None]`。
- `run_job` を拡張し、既存の用途バケツ処理の後に品目バケツも処理する。品目分は `slug = f"{tone}#{cat_slug}"`、ランキングは (tone, cat_slug) のジャンルIDで取得。各 (tone, band) の処理結果から在庫ありの cat_slug を集めてマニフェストを書く。
- `curation.build_user_prompt` は現状 `CATEGORIES.get(slug)` で用途キーワードを引いてプロンプト文（「用途「○○」のお返し品…」）を作る。品目バケツでは「トーン（慶事/弔事）× 品目（例: タオル・寝具）のお返し品」を伝える分岐を足す。slug が `tone#cat` 形式かで判定し、品目ラベル＋トーン文言を差し込む。fit 評価指示（family/friend/work/other）はそのまま流用。
- `scoring.passes_gate(item, slug)` は用途スラッグ前提。品目スラッグでも安全に動くよう、品目バケツでは品目に応じた最小限のゲート（弔事カテゴリで慶事専用語を弾く等）に分岐するか、共通の害のないゲートに落とす。実装時に既存 `passes_gate` を読んで最小改修で対応する。

### スケジュール（`infra/cdk/lib/catalog-batch-stack.ts`）

- 現行は2ルール（Morning = UTC 20:00 = JST 5:00、Evening = UTC 8:00 = JST 17:00）の1日2回。これを 1本（JST 9:00 = UTC 0:00）の1日1回にする。コスト削減（Lambda 起動・LLM・楽天コールが半減）。
- 鮮度への影響: 価格は取得後23時間でマスクされ金額目安に落ちる（`adapter._MASK_AFTER`）。1日1回・JST 9:00 実行なら、当日中は鮮度内、翌8:00〜9:00 の約1時間だけ価格が金額目安表示になる。許容範囲。アイテム自体のTTLは48時間（`store._ITEM_TTL`）で1日1回でも期限切れにはならない。
- docstring の「JST 5:00/17:00」を「JST 9:00」に更新。

## フロントエンド（`frontend/src/App.tsx` の `suggest` 画面）

- 画面上部にタブ（チップ）の横スクロール行を追加。先頭固定で「おすすめ」、続けて API の `categories` を描く。トーンは API 側で絞って返すので、フロントは受け取ったまま出せばよい。
- 選択中タブの状態 `activeCategory: string | null`（null = おすすめ）を持つ。タップで `api.suggestions(..., category)` を呼び `suggestions` を差し替え。読み込み中は軽いプレースホルダ。
- 既存のカード描画（画像・評価・価格・理由・「商品を見る ↗」・「この品に決める」）はそのまま品目タブでも使う。
- クリック計測（`api.clickSuggestion`）は現行のまま（bucket は item 由来なので品目バケツでも整合）。
- アフィリエイト免責・「価格は変動します」フッターは現行どおり常時表示。
- コピーは既定スタイル（太字・装飾絵文字を使わない）。タブのラベルはタクソノミの表示名をそのまま使う。

## 楽天 API 規約（既存方針を踏襲・変更なし）

- 価格・在庫キャッシュ24時間（マスク23h）、その他3ヶ月。取得日時＋免責表示は維持。
- 収益化は楽天アフィリエイト経由のみ。`affiliate_url` をそのまま使う。
- レート約1req/秒。品目追加でコール数が増える点はコスト/性能リスクに記載。

## コスト・性能リスクと緩和

- リスク: バケツ 63→147（約2.3x）。楽天検索（各バケツ2頁）＋ランキング＋LLM が増え、単一Lambda 15分（`timeout: Duration.minutes(15)`）を超える恐れ。
- 1日2回→1回でLLM・楽天の総コストは時間当たり半減するが、1回あたりの処理量（バケツ数）は増える点に注意。15分バジェットは「1回あたりの所要時間」の問題。
- 緩和（実装時に計測してから判断、強い順）:
  1. まず実測。`run_job` には既に deadline ベースの LLM 打ち切り（残り3分で線形フォールバック）があるため、超過してもクラッシュせず線形で確定はする。品目バケツがフォールバック多発するなら次の手を打つ。
  2. トーンで2回に分割（慶事ジョブ / 弔事ジョブを別 EventBridge ルール・別時刻に）。単一Lambda 15分を維持したまま並列度を上げずに1回あたりを軽くできる。
  3. 品目を段階導入（まず非食品の受け皿 towel/tableware/catalog/daily を優先投入し、効果を見て残りを足す）。
  4. それでも足りなければ Step Functions 化（正本設計で v2 と整理済み。今回は持ち込まない）。
- 監視: 既存の `CatalogJobBucketsFailed` / `CatalogLlmFallbackCount` / `CatalogFitDegenerationCount`（EMF）。品目バケツも同じ集計に乗る。閾値超過を当面は手動で確認する。

## テスト観点

- `adapter`: `category` 指定で `BUCKET#<tone>#<cat>#<band>` を引くこと／無指定は現行パスのままであること（回帰）／隣接補完・鮮度マスク・0件フォールバックが品目でも効くこと。
- `adapter.available_categories`: マニフェストを読んでトーン別の表示名つきで返すこと／マニフェスト無し（旧データ）では空を返し画面が「おすすめ」だけで成立すること。
- `buckets`: `slug_of` 等の既存挙動が不変であること／品目スラッグの組み立て（`f"{tone}#{cat}"`）。
- `job.run_job`: 品目バケツが処理されマニフェストが書かれること／用途バケツの既存テストが不変であること（回帰）。
- `curation.build_user_prompt`: 品目スラッグでトーン＋品目の文面が入り、用途スラッグでは現行文面が不変であること。
- フロント: タブ切替で `api.suggestions` が `category` 付きで呼ばれること／`categories` 空ならタブが「おすすめ」のみになること／トーンに応じたチップが出ること。

## 移行・互換

- 完全に非破壊の追加。既存バケツ・既存配信・既存画面（category 無指定）は不変。
- 初回は品目バケツ・マニフェストが空のため、`categories` は空で返り、画面は現行どおり「おすすめ」だけで成立する。バッチが一度走ると品目タブが現れる（初回は手動起動でシード、正本設計の運用と同じ）。
- ロールバックは品目バケツ・マニフェストを無視するだけで成立（データは48hでTTL消滅）。

## 未確定・実装時に確定する事項

- 各品目の楽天ジャンルID（上位最頻ジャンルを実機採取）。
- `passes_gate` の品目対応の具体（既存実装を読んで最小改修）。
- タブのラベル最終文言（表示名は上表ベース、長い「飲料（コーヒー・日本茶・紅茶）」はチップでは「飲料」に短縮し詳細は注記、など実装時に画面で調整）。
