# レビュー方針（REVIEW.md）

noshi は家族・親族の贈答（もらった／あげた）を管理する Web プロダクト。
指摘は**日本語**で書く。

## 🔴 Important の定義（noshi 固有）

次に該当するものだけを Important とする:

- **本人／世帯スコープ漏れ**（OWASP A01）。他人・他世帯のデータに触れ得る
  DynamoDB アクセスや API エンドポイント。
- **内部情報の漏洩**: エラー応答やログに stack trace・内部 ID・PII（メール／氏名等）が
  出ている。汎用文言になっていない。
- 本番を壊すロジック誤り、後方非互換なデータ移行・API 変更。
- 秘密情報（鍵・トークン）のコミット。

スタイル・命名・リファクタ提案は Nit 止まり。

## 必ず確認する

- すべての永続化アクセス／API が**呼び出しユーザーの本人・世帯に絞られている**か。
- エラーは**汎用文言**で内部情報を漏らさないか。
- 新しいドメイン／サービスのロジックや bugfix に**テストが伴っているか**（TDD 必須）。
  テスト名が「何を検証するか」の日本語一文になっているか。
- frontend で**ネイティブ `<select>` 等を使っていない**か
  （デザインシステム: `Select`／`MasterSelect`／`PartySelect` の `rel-panel` パターン）。
  a11y は `<button>`／`fieldset`／`aria-*` を適切に。

## 報告しない（ノイズ抑制）

- CI が担保するもの: **ruff／biome／mypy**（lint・format・型）。
- 生成物: `cdk.out*`／`dist/`／`.venv/`／`node_modules/`。
- 単なる整形の好み。

## 出し方

- 指摘は**件数を絞らず**すべて出す（レビューは PR 作成時の一度きりのため）。
- 挙動の主張は **`file:line` で裏取り**してから出す（命名からの推測で出さない）。
- 要約は冒頭に一行サマリ（例: `重大1・軽微4`）。問題なければ「ブロッカーなし」から始める。
