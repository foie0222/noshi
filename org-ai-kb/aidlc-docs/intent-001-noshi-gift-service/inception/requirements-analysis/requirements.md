# Requirements — noshi（MVP）

## Intent summary

- **Type:** prototype（新規プロダクトの構想・設計）
- **Scope:** 単一サービス（システム全体で1サービス、MVP は個人ソロ利用）
- **Complexity:** 中（生成AI＋OCR/AI抽出を含むが、MVP 範囲では単一ユーザー・単一データ境界）
- **Classification:** greenfield（既存コードなし）
- **Affected repos:** none
- **製品定義:** 家族・親族・友人との贈答（もらった/あげた）を AI が一元管理する生成 AI Web プロダクト。ご祝儀袋等を撮影 → 抽出・記録 → 半返し計算 → お返し提案・礼状生成 まで支援する。
- **MVP 対象ユーザー:** 贈答（慶弔・季節の贈り物）の記録と適切なお返しに悩む個人。1人で自分の贈答を管理する。

## Functional requirements

各要件は pass/fail で検証可能な単位で記述する。

- **FR-1 アカウント/認証**
  - FR-1.1 ユーザーは外部 IdP もしくはメールでサインアップ・ログインできる。
  - FR-1.2 各ユーザーのデータは本人のみがアクセスでき、他ユーザーのデータは一切閲覧・取得できない（最小権限）。
  - FR-1.3 ユーザーは自分のアカウントと全データの削除を要求でき、削除が完了する。

- **FR-2 撮影 → 記録（AI 抽出）**
  - FR-2.1 ユーザーは贈答に関する画像（ご祝儀袋・のし・贈答品・送り状等）を1枚以上アップロードできる。
  - FR-2.2 システムは画像から候補項目を抽出する: 金額、相手の氏名、続柄/関係、用途（結婚・出産・香典・お中元等）、日付。
  - FR-2.3 抽出結果はユーザーが確認・修正してから保存される（AI 抽出を確定前提にしない）。
  - FR-2.4 画像が判読不能・抽出失敗の場合、手入力での記録にフォールバックできる。

- **FR-3 Give 履歴台帳**
  - FR-3.1 「もらった」「あげた」の贈答レコードを作成・編集・削除できる。各レコードは相手・金額・用途・日付・方向（受領/贈与）・メモを持つ。
  - FR-3.2 履歴を相手別・用途別・期間別に一覧・検索・集計できる。
  - FR-3.3 相手ごとに「もらった合計 / あげた合計 / 差分」を表示できる。

- **FR-4 半返し計算**
  - FR-4.1 もらった金額と用途から、推奨お返し額（レンジ）を算出する。
  - FR-4.2 算出根拠（適用したルール: 例「香典は半返し」「出産祝いは1/3〜半返し」等）を提示する。
  - FR-4.3 ユーザーは推奨額を上書きでき、上書き値が以後の提案・記録に反映される。

- **FR-5 お返し品提案（MVP は提案のみ）**
  - FR-5.1 予算（半返し額）・関係・用途から、お返し品の候補を提示する。
  - FR-5.2 各候補は概要と外部参照リンクを持つ（MVP では noshi 内で購入・決済は行わない）。
  - FR-5.3 ユーザーは候補から1つを「これにする」と選択し、贈答イベントに紐づけて記録できる。

- **FR-6 礼状文面生成**
  - FR-6.1 相手・用途・関係・トーン（丁寧/カジュアル等）を指定すると、礼状/メッセージ文面を生成する。
  - FR-6.2 生成文面はユーザーが編集でき、コピー/書き出しできる。

- **FR-7 贈答イベント管理**
  - FR-7.1 受領レコードに対し「受領 → お返し検討中 → 完了」のステータスを管理できる。
  - FR-7.2 お返し未完了のイベントを一覧で確認できる（やり残し防止）。

## Non-functional requirements

測定可能な目標として記述する（実現手段=技術選定は construction フェーズ）。

- **NFR-1 性能**
  - NFR-1.1 主要画面の初回表示は p95 < 2.5s（モバイル回線想定）。
  - NFR-1.2 画像からの AI 抽出は p95 < 10s で結果（または進捗）を返す。

- **NFR-2 セキュリティ/プライバシー（OWASP lens）**
  - NFR-2.1 データ分類: 相手の氏名・住所・続柄・贈答金額は **confidential〜restricted** に分類。分類に応じた取り扱いを必須とする。
  - NFR-2.2 保存時暗号化（at rest）・転送時 TLS（in transit）を必須とする。restricted データを平文で保存・ログ・送信しない。
  - NFR-2.3 アクセス制御（OWASP A01）: 全データアクセスは本人のスコープに限定。他者データへの参照は設計上不可能であること。
  - NFR-2.4 認証（A07）: 認証失敗のレート制限、セッション/トークンの有効期限と失効を定義。
  - NFR-2.5 入力検証（A03）: 画像アップロード・フォーム・生成プロンプト等、外部入力境界で検証する。エラー応答は内部情報（スタックトレース・パス等）を漏らさない。
  - NFR-2.6 監査ログ: 認証試行・認可失敗・データ削除・エクスポート等のセキュリティ関連操作を追跡できるログを残す。
  - NFR-2.7 生成AI/外部送信: AI 抽出・文面生成で外部 LLM に送るデータを最小化し、restricted データの外部送信時は同意・最小化・マスキングを検討する。

- **NFR-3 可用性・スケール**
  - NFR-3.1 MVP は単一リージョンで月間稼働率 99% を目標。

- **NFR-4 ユーザビリティ/アクセシビリティ**
  - NFR-4.1 モバイルファーストのレスポンシブ Web（PWA を視野）。主要操作は片手・3タップ以内を目標。
  - NFR-4.2 主要画面は WCAG 2.1 AA 相当のコントラスト・代替テキストを満たす。

- **NFR-5 コンプライアンス**
  - NFR-5.1 個人情報保護法（APPI）に準拠。利用目的の明示と同意取得。
  - NFR-5.2 第三者（贈答相手）の PII を本人が登録する構造に対し、取り扱い方針・削除権・漏洩時対応を定義する。

## Assumptions

以下は前提（事実ではなく assumption として明示）:

- MVP では noshi 内決済を行わない（お返しは提案のみ）。
- ネイティブアプリは提供せず Web のみ。
- データ共有（家族・親族）は行わず、単一ユーザーのプライベート台帳とする。
- 認証は外部 IdP 利用を第一候補とする（最終決定は construction）。
- AI 抽出・文面生成は外部 LLM/OCR の利用を想定（ベンダー選定は construction）。
- 金額・用途の半返しルールは日本の一般的な慣習を初期値とし、後で調整可能とする。

## Out of scope（本 intent では除外）

- ⑥ お返し品の発送承認・物流・配送代行
- ⑦ お年玉の年齢別相場
- ⑧ 贈与税 110 万円枠の管理
- ⑨ 親族間の贈答バランス警告
- 家族世帯・親族グループでの共有/共同管理
- EC 連携による noshi 内購入・決済
- ネイティブモバイルアプリ
