# 自律オペレーター（背骨 + PO ゲート + デリバリーループ）設計

作成日: 2026-06-27
ステータス: 設計合意済み（実装計画はこのあと writing-plans で作成）

## 目的

noshi の開発・運用を Claude で自律化し、最終的にリリースと収益化（楽天アフィリエイト）に
責任を負わせる構想の、その土台となる最初のサブプロジェクト。

オーナー（人間）はプロダクトオーナー（PO）として承認ゲートだけに残り、Claude が
デリバリーチーム（エンジニア + 運用 + 提案役）として自走する。これは形式的には
Level 4（Approver）自律に相当する。Level 5（完全自律）は事前調査で信頼できる実例が
存在せず、収益を自律運営した事例も皆無（唯一の正面実験は売上ゼロ）だったため、
本設計では意図的に Level 4 に留める。

### 目的関数

実ユーザー獲得 → お返し品提案の利用 → 楽天アフィリエイト収益。
現在は実ユーザーゼロのため、当面の最大レバーは「ローンチ可能な品質」と「集客準備」。
監視・障害対応はトラフィックが出てから比重が上がる（本サブプロジェクトの後段で扱う）。

## スコープ

### このサブプロジェクトで作るもの

GitHub ネイティブの自律デリバリーループ:
承認済み Issue → 実装 → テスト → PR → 自動レビュー（既存）→ ハイブリッドマージ →
本番デプロイ（既存）。加えて、提案生成・状態管理・PO ブリーフィング。

### このサブプロジェクトで作らないもの（後続サブプロジェクト）

2. システム・テレメトリと自己修復（ヘルスチェック・アラーム → Issue 化 → 修正PR）
3. 事業 KPI 計測と PO ブリーフィングの高度化
4. Growth 実行（外向き・PO ゲート厳格、リリース後に本格化）

## 役割分担（境界）

Claude が自走（承認不要）:
- 承認済み項目の実装 → PR → CI 通過 → （ハイブリッド）マージ → 本番デプロイ
- 運用: 監視・障害検知・原因調査・修正PR・復旧（後続サブプロジェクトで本格化）
- 保守: 依存更新の追従、リファクタ、テスト追加
- 提案: バックログ案・優先度案・改善案を作る（採用は PO）

PO（人間）が承認:
- 何を作るか/作らないか、優先順位の確定（`po:approved` ラベル付与）
- リリースの Go/No-Go
- 外向き全般（公開発信・ユーザー接触・ブランド/コピー）
- カネ（課金設計・価格・広告出稿・有料ツール契約）
- 収益/アフィリエイト戦略の方針変更
- センシティブな変更の本番マージ（後述のマージゲート）

## アーキテクチャ

実行基盤は GitHub ネイティブ。状態はすべて GitHub Issue のラベルに永続化する
（ランナーは使い捨てのため、ランナー内に状態を持たない）。認証は既存の Max OAuth
トークン（`secrets.CLAUDE_CODE_OAUTH_TOKEN`）で定額・追加課金なし。

既存資産（変更しない、乗るだけ）:
- `ci.yml`: backend/frontend/infra の lint・型・テスト、main push で本番自動デプロイ（OIDC）
- `claude-review.yml`: PR 作成時に Claude 自動レビュー（インラインコメント、未解決でマージ不可）

### バックログの状態機械（ラベル）

- `po:proposal`: Claude が目的関数から生成した提案。Claude は実装しない。PO の判断待ち。
- `po:approved`: PO が Go。Claude が実装に着手する唯一のトリガ。
- `po:rejected`: 却下（または Issue クローズ）。
- `agent:in-progress`: オペレーターが着手中。二重着手防止の排他ロック。
- `agent:pr-open`: PR を開いた状態。
- `needs:po`: 実装中に PO 級の判断（外向き・カネ・方向性）にぶつかり、手を止めて PO に質問。

### マージゲート（リスク別ハイブリッド）

PR の変更ファイル・差分サイズ・ラベルから自動判定し、`merge:auto` か `merge:human` を付与。
マージ用ワークフローは「CI 緑 ＋ 自動レビュー通過 ＋ `merge:auto`」のときだけ squash 自動マージ。

自動マージ可（低リスク）:
- 依存更新（Dependabot、CI ゲート済み）
- ドキュメント・コメント・テストのみの変更
- UI 文言・デザインシステム内のスタイル調整
- しきい値以下の小差分で、下記センシティブパスに触れないもの

人間（PO）マージ必須（センシティブ）:
- カネ/アフィリエイトに触るパス（楽天アフィリエイト・お返し品提案・将来の課金）
- 認証・本人/世帯スコープ（OWASP A01 のスコープ判定ロジック）
- インフラ（`infra/cdk/**`）・DB スキーマ/マイグレーション
- 外向きコンテンツ（公開コピー・メール・マーケ文言）
- しきい値超の大差分（レビューが分割要求）
- `needs:po` が立った Issue 由来の PR

判定の仕組み: センシティブパスの glob リスト ＋ 差分行数しきい値 ＋ ラベル上書き。
正確な glob は実装時にコードを読んで確定する（カテゴリは上記6つで固定）。

## ワークフロー構成（新規3本）

### operator.yml（cron 1日1回 + 手動 dispatch）

1. Sense: `gh` で Issue をラベル集計、直近の CI/デプロイ結果、Dependabot 失敗を収集。
2. Act: `po:approved` の最優先1件に `agent:in-progress` を付与し、implement をディスパッチ。
3. Propose: `po:approved` が枯れていたら、目的関数から提案を `po:proposal` Issue 化（1日 N 件上限）。
4. Report: ブリーフィングを固定 Issue にコメント追記。

### implement.yml（`po:approved` 付与 or operator からの dispatch で起動）

- claude-code-base-action で ブランチ → 実装 + テスト → PR（本文に `Closes #N`）→ `agent:pr-open`。
- 着手前に AGENTS.md / REVIEW.md / CLAUDE.md を読ませ規約順守。
- 失敗時は Issue にエラーを残し `needs:po` を付与（または上限回数まで再試行）。
- 1 Issue = 1 PR の単発。ループさせない（Ralph 型は曖昧/設計案件に不向きと公式明言のため不採用）。

### classify-and-merge.yml（PR の CI 緑 ＋ レビュー通過で起動）

- 変更パス・差分サイズから `merge:auto` / `merge:human` を判定・ラベル付与。
- `merge:auto` のみ squash 自動マージ（→ 既存 deploy が発火）。`merge:human` は通知のみ。

## 安全柵とコスト

事前調査で確認された落とし穴（エージェントコードの冗長性・技術的負債、レビュー過信、
ループのトークン爆発、CI 失敗がエージェントに戻らない、レビュー量で時間が溶ける）を反映。

- 反復制御: `--max-turns` 設定。implement は単発、ループ禁止。
- 物理的レート制限: operator は1日1回、implement は同時1件、提案は1日 N 件まで。
- コスト: Max OAuth トークンで定額（追加課金なし）。暴走は上記レート制限で抑止。
- 品質ガード: 大差分は分割要求（Intercom 方式）、重複・カバレッジ低下を CI で検出、
  センシティブパスは人間マージ。
- セキュリティ: ワークフローのプロンプトインジェクション対策（Issue 本文を信頼しない、
  許可ツールを最小化）。秘密情報は SSM/secrets のまま。

## オペレーターの観測とブリーフィング

- 観測項目: オープン Issue（ラベル別）、直近デプロイの成否、CI 失敗、Dependabot 滞留
  （段階2以降: エラー率・コスト・KPI）。
- ブリーフィング（固定 Issue に追記）: 昨日やったこと / 今のオープン PR /
  承認待ち `po:proposal` 一覧 / 判断が要る `needs:po` / 数値サマリ。
- PO の操作は「Issue を見て承認ラベルを付ける」「センシティブ PR のマージを押す」だけで完結。

## 設計判断の根拠（事前調査より）

- Level 4（Approver）に留める: 信頼できる全本番事例（Elastic / Intercom / Classmethod /
  Zenn 個人開発者）が auto-merge を意図的に無効化し、人間をマージゲートに残していた。
- GitHub ネイティブ・cron→Issue→PR: Zenn の個人開発者が同型ループを実証（直接転用可能）。
- ハイブリッドマージ: Intercom の「大きい PR は分割要求」「説明責任とロールバックは人間」に倣う。
- 収益責任は段階分離: 収益を自律運営した信頼できる事例は皆無（AI CEO 実験は売上ゼロ）。
  前例のある「実装・運用・施策の実装と計測」までを自走化し、「収益を生む外向き判断」は
  Claude が提案・PO が承認、という現実的分担にする。

## 未解決事項（実装計画フェーズで詰める）

- センシティブパスの正確な glob（コードを読んで確定）。
- 差分サイズしきい値の具体値。
- 提案生成の1日上限 N の具体値。
- operator の cron 時刻（Zenn 例は平日3時、週末スキップ）。
- 既存ブランチ運用（`issue-<番号>` / squash merge）との整合確認。
