# User Stories — noshi（MVP）

ジャーニー順。各ストーリーは AC（pass/fail 検証可能）と Requirements トレースを持つ。INVEST 準拠。
横断 AC（全ユーザー向けストーリーに適用）: 性能 NFR-1（主要画面 p95<2.5s）、モバイルファースト/アクセシビリティ NFR-4。

## ジャーニー: 認証

### S-1 サインアップ/ログイン
As a 個人の贈答管理者, I want 外部 IdP かメールで安全にログインしたい, so that 自分の台帳に本人だけがアクセスできる。
- AC1: 外部 IdP もしくはメールでサインアップ・ログインできる。
- AC2: ログイン状態はセッションで維持され、ログアウトで失効する。
- Requirements: FR-1.1, NFR-2.4

### S-2 アカウント/データ削除
As a 個人の贈答管理者, I want アカウントと全データを削除依頼したい, so that 退会時に自分の機微情報が残らない。
- AC1: 削除要求ができ、完了後は本人データが参照不能になる。
- AC2: 削除操作は監査ログに記録される。
- Requirements: FR-1.3, NFR-2.6

## ジャーニー: 撮影 → 記録

### S-3 画像から贈答を記録
As a 個人の贈答管理者, I want ご祝儀袋等を撮影して内容を自動抽出したい, so that 手入力の手間なく正確に記録できる。
- AC1: 1枚以上の画像をアップロードできる。
- AC2: 金額・氏名・続柄/関係・用途・日付の候補が抽出され表示される。
- AC3: 抽出結果を確認・修正してから保存でき、未修正のまま確定はされない。
- AC4: 抽出失敗時は手入力にフォールバックできる。
- Requirements: FR-2.1, FR-2.2, FR-2.3, FR-2.4, NFR-2.5

## ジャーニー: Give 履歴台帳

### S-4 履歴の記録・閲覧・集計
As a 個人の贈答管理者, I want もらった/あげたを一覧・検索・集計したい, so that 相手ごとの収支を把握できる。
- AC1: 贈答レコードを作成・編集・削除できる（相手・金額・用途・日付・方向・メモ）。
- AC2: 相手別/用途別/期間別に一覧・検索できる。
- AC3: 相手ごとに もらった合計/あげた合計/差分 を表示できる。
- Requirements: FR-3.1, FR-3.2, FR-3.3

## ジャーニー: 半返し計算

### S-5 推奨お返し額の算出
As a 個人の贈答管理者, I want もらった額と用途から推奨お返し額を知りたい, so that 失礼のない金額で返せる。
- AC1: もらった金額・用途から推奨額（レンジ）が算出される。
- AC2: 適用ルールの根拠が提示される。
- AC3: 推奨額を上書きでき、上書き値が以後の提案・記録に反映される。
- Requirements: FR-4.1, FR-4.2, FR-4.3

## ジャーニー: お返し選択

### S-6 お返し品の提案と紐付け
As a 個人の贈答管理者, I want 予算・関係・用途に合うお返し品候補を見たい, so that 何を返すか迷わず決められる。
- AC1: 半返し額・関係・用途から品物候補が提示される。
- AC2: 各候補は概要と外部参照リンクを持つ（noshi 内決済はしない）。
- AC3: 候補を1つ選び、対象の贈答イベントに紐付けて記録できる。
- Requirements: FR-5.1, FR-5.2, FR-5.3, FR-7.1

## ジャーニー: 礼状

### S-7 礼状文面の生成
As a 個人の贈答管理者, I want 相手・用途・トーンに合う礼状文面を生成したい, so that 礼状作成の負担を減らせる。
- AC1: 相手・用途・関係・トーンを指定して文面を生成できる。
- AC2: 生成文面を編集し、コピー/書き出しできる。
- AC3: 生成のために外部 LLM へ送るデータは最小化される（restricted データの不要送信なし）。
- Requirements: FR-6.1, FR-6.2, NFR-2.7

## ジャーニー: 完了管理

### S-8 贈答イベントのステータス管理
As a 個人の贈答管理者, I want 受領→お返し検討中→完了 を管理したい, so that お返しのやり残しを防げる。
- AC1: 受領レコードに 受領/検討中/完了 のステータスを設定できる。
- AC2: お返し未完了のイベントを一覧で確認できる。
- Requirements: FR-7.1, FR-7.2

## システムストーリー（要のもののみ）

### S-9 AI 抽出サービスの挙動
As the noshi 抽出サービス, when ユーザーが画像を投入したとき, it must 候補項目を抽出し、信頼度が低い項目はユーザー確認を要求し、失敗時は手入力 fallback を返す。
- AC1: 抽出は p95 < 10s で結果または進捗を返す。
- AC2: 低信頼の項目は確認必須フラグ付きで返す。
- Requirements: FR-2.2, FR-2.3, NFR-1.2

### S-10 本人スコープの強制（アクセス制御）
As the noshi サービス, when いずれかのデータアクセスが発生したとき, it must 要求者本人が所有するデータに限定し、他者データへのアクセスを拒否する。
- AC1: ユーザーは自分が所有するレコードのみ取得・更新・削除できる。
- AC2: 他ユーザーIDを指定したアクセスは認可エラーになり、内部情報を漏らさない。
- Requirements: FR-1.2, NFR-2.3（OWASP A01）

## セキュリティ / abuse ストーリー

### S-11 他人データ参照の拒否（abuse）
As an attacker, when 他ユーザーのデータIDを指定して参照を試みたとき, the system must アクセスを拒否し記録する。
- AC1: 認可されないリソース参照は拒否され、汎用エラーを返す。
- AC2: 認可失敗は監査ログに記録される。
- Requirements: NFR-2.3, NFR-2.6（OWASP A01）

### S-12 認証の悪用防止
As an attacker, when ログインを総当たりで試みたとき, the system must レート制限し、セッション/トークンを失効管理する。
- AC1: 連続認証失敗にレート制限/ロックが働く。
- AC2: セッション/トークンに有効期限と失効がある。
- Requirements: NFR-2.4（OWASP A07）

### S-13 機微操作の監査証跡
As the noshi サービス, when 認証試行・認可失敗・データ削除・エクスポートが発生したとき, it must 監査ログを残す。
- AC1: 上記イベントが追跡可能な形でログ化される。
- AC2: ログに restricted データを平文で残さない。
- Requirements: NFR-2.6, NFR-2.2

## ジャーニー: プライバシー/同意

### S-14 利用目的の明示と第三者PIIの同意・分類
As a 個人の贈答管理者, I want 自分が登録する贈答相手の個人情報の扱いと利用目的を理解し同意したい, so that 安心して機微情報を預けられる。
- AC1: 初回登録時に利用目的が明示され、同意を取得する（APPI）。
- AC2: 登録するデータは機微度（confidential〜restricted）に応じて扱われることが分かる（データ分類の要件充足）。
- AC3: 第三者（贈答相手）の PII について、取り扱い方針・本人による削除の手段が提示される。
- Requirements: NFR-2.1, NFR-5.1, NFR-5.2

## カバレッジ補足（ストーリー化しない要件の扱い・理由）

Rule 4 に基づき、独立ストーリーにしない要件と理由を明記する。

- **NFR-1（性能）/ NFR-4（モバイルファースト・アクセシビリティ）:** 全ユーザー向けストーリー（S-1〜S-8）に適用する**横断的受け入れ条件**として扱う（明確化回答 Q4=ミックスに基づく）。冒頭「横断 AC」に明記済み。
- **NFR-3.1（可用性 99%）:** 運用/インフラの非機能目標であり、MVP 段階では discrete なユーザーストーリーにしない。construction フェーズの nfr-assessment / infrastructure-design で扱う**運用横断要件**として記録（意図的に非ストーリー化）。
