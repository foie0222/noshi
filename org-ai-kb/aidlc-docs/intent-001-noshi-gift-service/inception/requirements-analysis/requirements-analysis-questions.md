# Requirements Analysis — Clarification Questions

Intent: noshi（家族・親族・友人との贈答を AI が一元管理する生成 AI Web プロダクト）
Lens: OWASP active（セキュリティ/プライバシー観点を質問に注入）

スコープは Inception まで。MVP の輪郭を確定するための質問。

### Q1: MVP の最初の対象ユーザー

a) 個人ソロ利用（自分の「もらった/あげた」を一人で記録・管理）
b) 家族世帯で共有（夫婦・家族が同じ台帳を共同管理）
c) 親族グループ（親戚間でバランスを共有）

**Trade Offs:** 共有は価値が高いが、権限・プライバシー・招待フローが一気に複雑化する。ソロ起点なら最短で価値検証でき、共有は後続 intent で追加可能。

**Recommendation:** a) 個人ソロ利用から。

[Answer]: a) 個人ソロ利用から

### Q2: MVP のコア機能（含めるものを選択）

候補: ①ご祝儀袋/贈答品の撮影→記録（OCR/AI抽出: 金額・氏名・関係・用途）、②もらった/あげた Give 履歴台帳、③半返し計算、④お返し品の提案、⑤礼状文面の生成、⑥発送承認/発送、⑦お年玉の年齢別相場、⑧贈与税110万円枠の管理、⑨親族間バランス警告

**Trade Offs:** ①②③が「記録して続けられる」中核ループ。④⑤は生成AIの差別化価値だが外部連携/品質担保が要る。⑥は物流連携で重い。⑦⑧⑨は付加価値で後回し可能。

**Recommendation:** MVP = ①②③（中核ループ）＋④⑤（提案・礼状は生成のみ）。⑥⑦⑧⑨は次 intent。

[Answer]: ①撮影→記録, ②Give履歴台帳, ③半返し計算, ④⑤提案・礼状生成(AI) を MVP に含む。⑥発送/⑦お年玉相場/⑧贈与税枠/⑨バランス警告 は次 intent

### Q3: 「お返し品提案・発送」の実装深度

a) 提案のみ（品物候補をリスト/リンク表示。購入は外部で）
b) EC 連携で購入まで（Amazon/楽天等のアフィリエイト or API）
c) 発送代行まで（noshi が配送を手配）

**Trade Offs:** (a) は最小で価値検証可。(b)(c) は外部API・与信・物流・返品対応など非機能/法務が急増。

**Recommendation:** a) MVP は提案のみ。

[Answer]: a) 提案のみ

### Q4: プライバシー/データ保護モデル（OWASP lens）

扱うデータ: 贈答相手の氏名・関係性・金額・住所・用途（= confidential〜restricted の PII）。

a) 本人のみ閲覧・保存時暗号化・最小権限（共有なし）
b) 家族共有を許可（共有範囲の同意管理が必要）
c) こだわらない

**Trade Offs:** noshi は他者（贈り先）の個人情報を本人が登録する構造。第三者 PII の取り扱い同意・削除権・漏洩時影響が論点（APPI）。

**Recommendation:** a) MVP は本人のみ・暗号化・最小権限。共有は将来。

[Answer]: a) 本人のみ閲覧・保存時暗号化・最小権限

### Q5: プラットフォーム / 体験の前提（確認）

撮影が起点のため **モバイルファーストのレスポンシブ Web** を想定。ネイティブアプリは将来。

**Recommendation:** モバイルファースト Web（PWA 視野）。

[Answer]: モバイルファースト レスポンシブ Web（PWA 視野）。推奨値を採用（人間の異議なし）

### Q6: MVP の成功指標（ビジネス文脈）

何をもって「使えるサービス」とみなすか。例: 月内の贈答イベント登録数 / 半返し計算の利用率 / 翌イベントでの継続率。

**Recommendation:** 「贈答イベントを登録し、半返し計算まで到達したユーザーの継続率」を主指標に仮置き。

[Answer]: 主指標=「贈答イベントを登録し半返し計算まで到達したユーザーの継続率」。推奨値を採用（人間の異議なし）
